## Part 1: 当前状态全景记录

### 1.1 快速录入（Workflow 1 / QuickInput）

#### 1.1.1 产品目标

快速录入的核心定位是**"最低摩擦路径"**——让用户用最少操作完成个人主页创建。

| 目标编号 | 目标描述 | 验证指标 |
|---------|---------|---------|
| H1 | 用户更愿意「口述录入」而非「手动填表」 | 口述路径完成率 ≥ 70% |
| H3 | AI 提取的核心信息足够准确 | 用户实质性纠偏率 ≤ 30% |
| — | 一段文字输入 → 结构化主页，用户只需轻量编辑即可发布 | 主观体验评估 |

设计哲学：**"用户是 Boss"** — AI 负责 draft，用户拥有绝对确认/拒绝/编辑权。

#### 1.1.2 完整前端流程

```
用户输入文本 ──► [点击「生成主页」]
                    │
                    ▼
            handleGenerate()
                    │
                    ▼
            POST /api/chat
            ├── systemPrompt: state.config.promptExtractTags
            ├── userPrompt: state.quickInput（用户输入的完整文本）
            ├── model: state.config.model（默认 Doubao-Seed-2.0-lite）
            ├── temperature: state.config.temperature（默认 0.7）
            └── responseFormat: { type: 'json_object' }
                    │
                    ▼
            AI 返回 { content: JSON字符串 }
                    │
                    ▼
            JSON.parse(data.content)
            ├── blocks: [{ category, content }] → quickBlocks
            ├── tags: [{ text, type }] → quickTags
            └── intro: string → quickIntro
                    │
                    ▼
            前端展示 editable 结果
            ├── 简介文本（可编辑）
            ├── 标签云（可删除、不可新增）
            └── 文本块列表（可编辑、可删除）
                    │
                    ▼
            [点击「确认并发布到个人主页」]
                    │
                    ▼
            publishToProfile()
            ├── profile.intro = quickIntro
            ├── profile.tags = quickTags
            ├── profile.blocks = quickBlocks
            └── activeTab = 'profile'
```

**关键代码路径：** `components/workflow1/QuickInput.tsx`

- `handleGenerate` 函数（第 31-82 行）：核心调用逻辑
- `updateBlock` / `deleteBlock` / `deleteTag`：用户编辑接口
- `publishToProfile`（第 105-115 行）：发布逻辑

#### 1.1.3 提示词全文

**`promptExtractTags`（标签提取 & 内容分块 Prompt）：**

```
你是一位擅长人物侧写的编辑。请根据用户的自我介绍文本，提取结构化信息。

要求：
1. 将内容按维度拆分为文本块（self_intro/background/offer/need/custom）
2. 提取 4-10 个标签，每个标签标注类型（belong/offer/need/follow）
3. 生成一段 100-200 字的个人简介

输出严格 JSON 格式：
{
  "blocks": [{"category": "self_intro|background|offer|need|custom", "content": "文本内容"}],
  "tags": [{"text": "标签", "type": "belong|offer|need|follow"}],
  "intro": "简介文字"
}
```

**`promptGenerateIntro`（简介生成 Prompt）：**

```
基于以下用户信息，生成一段 100-200 字的个人简介。语言简洁、有温度，体现个人特质。只输出简介文字，不输出其他内容。
```

#### 1.1.4 设计意图分析

**为什么用这 5 个 category？**

| Category | 含义 | 设计意图 |
|---------|------|---------|
| `self_intro` | 自我介绍 | 用户主动呈现的核心身份 |
| `background` | 历史背景 | 教育、职业经历等溯源信息 |
| `offer` | 能提供的帮助 | **互赖网络核心** — 校友能提供什么资源/能力 |
| `need` | 具体需求 | **互赖网络核心** — 校友需要什么帮助 |
| `custom` | 自定义 | 兜底类别，容纳无法归类但用户想展示的内容 |

`offer` 和 `need` 是同学录网络的价值核心——它们直接驱动校友之间的匹配和互助。这是从第一代"基地"产品中继承的设计。

**为什么用这 4 个 tag type？**

| Tag Type | 含义 | 匹配逻辑 |
|---------|------|---------|
| `belong` | 归属标签 | "实验中学2006届""清华计算机系" — 用于群体归属和筛选 |
| `offer` | 提供标签 | "AI创业""产品设计" — 用于能力供给侧匹配 |
| `need` | 需求标签 | "寻找技术合伙人""海淀咖啡探店" — 用于需求侧匹配 |
| `follow` | 关注标签 | "马拉松""科幻小说" — 用于兴趣社交匹配 |

`offer` ↔ `need` 的匹配是产品最核心的价值主张。

**为什么要求 JSON 格式？**

结构化输出是前端解析的前提。`{ type: 'json_object' }` 强制模型返回合法 JSON，减少解析失败。但当前 prompt 中没有容错说明（如"如果无法解析则返回最简化结构"）。

**为什么简介限定 100-200 字？**

这是个人主页"摘要"的合理长度——足够传达核心信息，又不至于让读者失去耐心。对应 UI 中简介区域的高度设计。

### 1.2 AI 采访（Workflow 2 / InterviewChat）

#### 1.2.1 产品目标

AI 采访的核心定位是**"渐进式挖掘"**——通过对话降低用户填写门槛，同时挖掘用户自己没想到的信息。

| 目标编号 | 目标描述 | 验证指标 |
|---------|---------|---------|
| H2 | AI 采访能挖掘出用户自己没想到的标签 | AI 新增标签占比 ≥ 30% |
| — | 收集 hidden 信息（用户不愿公开但希望 AI 匹配时考虑） | hidden 信息质量评估 |
| — | 用对话降低门槛（比一次性写大段文字更轻松） | 采访完成率 |

#### 1.2.2 完整前端流程

```
页面加载 ──► useEffect
                │
                ▼
        如果 interviewMessages.length === 0 且未结束
                │
                ▼
        startInterview() ──► POST /api/chat
                │           systemPrompt: promptInterviewGuide + "\n\n这是采访的第 1 轮..."
                │           userPrompt: "开始采访"
                │           responseFormat: { type: 'json_object' }
                ▼
        AI 返回 JSON:
        {
          "nextQuestion": "AI 的第一个问题",
          "hiddenBlocks": [],
          "extractedTags": [],
          "introSnippet": ""
        }
                │
                ▼
        展示 AI 问题 ←────── 用户输入回答 ──► [点击「发送」]
                │                              │
                │                              ▼
                │                      handleSend()
                │                              │
                │                              ▼
                │                      POST /api/chat
                │                      systemPrompt:
                │                        promptInterviewGuide
                │                        + "\n\n当前是第 N 轮（最多 6 轮）\n\n历史对话：\n{history}"
                │                      userPrompt: 用户回答
                │                              │
                │                              ▼
                │                      AI 返回 JSON:
                │                      {
                │                        "nextQuestion": "...",
                │                        "hiddenBlocks": [{"content": "..."}],
                │                        "extractedTags": [{"text": "...", "type": "..."}],
                │                        "introSnippet": "..."
                │                      }
                │                              │
                │                              ▼
                │                      前端更新：
                │                      ├── interviewMessages 追加用户回答 + AI 问题
                │                      ├── interviewBlocks 追加 hiddenBlocks
                │                      ├── interviewTags 追加 extractedTags
                │                      ├── interviewIntro = introSnippet
                │                      └── interviewRound += 1
                │                              │
                │                              ▼

                │
                ▼

                │

                │

        ├── systemPrompt: promptExtractTags
        │                 + "\n\n注意：请同时基于以下 hidden 信息提炼..."
        │                 + hiddenContext
        └── userPrompt: "请根据以下采访对话，生成完整的个人主页内容：\n{history}"
                │
                ▼
        AI 返回 JSON { blocks, tags, intro }
                │
                ▼
        更新 interviewBlocks（替换非 hidden 部分）

        更新 interviewIntro

                │
```

#### 1.2.3 提示词全文

**`promptInterviewGuide`（采访引导 Prompt）：**

```
你是一位亲切的校友采访者，正在帮助实验中学2006届校友整理个人档案。

采访策略：
1. 每次只问一个问题，语言自然亲切
2. 根据用户回答追问细节，挖掘深层标签
3. 覆盖四个维度：自我介绍、历史背景、能提供的帮助、具体需求
4. 最多追问不超过限制轮数

同时，从用户的回答中提取 hidden 文本块——这些是用户可能不愿意公开展示、但希望 AI 在匹配搜索时能了解到的信息（如真实诉求、敏感背景、深层需求等）。

输出严格 JSON 格式：
{
  "nextQuestion": "下一个问题",
  "hiddenBlocks": [{"content": "隐藏的深层信息"}],
  "extractedTags": [{"text": "标签", "type": "belong|offer|need|follow"}],
  "introSnippet": "基于目前信息的简介片段"
}
```

#### 1.2.4 设计意图分析

**"每次只问一个问题"**

降低用户的认知负荷。一次性抛出多个问题会让用户感到压力，可能选择性地忽略某些问题。单问题策略确保每个回答都聚焦。

**"hidden 文本块"机制**

这是产品的差异化设计。用户可能不愿意在公开主页上写"我正在秘密筹备创业"或"我想离开现在的婚姻"，但这些信息对 AI 匹配合适的校友帮助极大。Hidden 机制实现了"隐私分级"：
- 公开层：profile 中展示的内容
- 隐藏层：仅 AI 在匹配搜索时可见，不对外展示

**四个维度的覆盖策略**

与快速录入的 5 个 category 对应：
- 自我介绍 → `self_intro`
- 历史背景 → `background`
- 能提供的帮助 → `offer`
- 具体需求 → `need`

**为什么同时输出 4 个字段？**

| 字段 | 用途 | 实时性 |
|------|------|--------|
| `nextQuestion` | 驱动对话继续 | 每轮必需 |
| `hiddenBlocks` | 累积 hidden 信息 | 增量追加 |
| `extractedTags` | 实时提取标签 | 增量追加 |
| `introSnippet` | 给用户"进度感" | 每轮更新 |

`introSnippet` 的设计很巧妙——即使采访还没结束，用户也能看到"基于目前信息，你的简介可能是..."，这给用户即时的反馈和继续配合的动力。

**MAX_ROUNDS = 6 的设定**

6 轮 ≈ 3-6 分钟对话时间。既足够挖掘信息，又不会让用户感到冗长。但这个数字是**经验值**，没有信息饱和度的动态判断。

## Part 2: 现状问题深度诊断

### 2.1 快速录入问题

#### 问题 1：无 Few-shot 示例

**问题描述**：Prompt 中只有指令，没有输入→输出的示例。AI 对"文本块"的理解可能不稳定。

**具体表现**：
- 同样一段输入，多次调用可能得到不同数量的 blocks
- AI 可能把一段完整的话拆成过细的碎片，或合并成过于粗粒度的块
- `category` 分类可能不一致（如把"在字节做产品"分到 `background` 还是 `self_intro`）

**对产品目标的负面影响**：
- 用户需要花更多时间编辑/重组 blocks（纠偏率上升，违反 H3）
- 标签质量不稳定，影响后续匹配准确性

#### 问题 2：无「标签挖掘」指令

**问题描述**：Prompt 只要求"提取标签"，没有明确要求 AI 从字里行间推断**隐含标签**。

**具体表现**：
- 用户说"我在字节做了3年产品，现在出来做 AI 创业"
- AI 可能只提取「字节」「产品经理」「AI创业」3 个标签
- 但隐含可提取的还有：「互联网大厂经验」「技术转商业」「连续创业者」「SaaS/工具方向」等

**对产品目标的负面影响**：
- AI 新增标签占比可能低于 30%（违反 H2）
- 用户主页标签过于"字面化"，匹配潜力不足

#### 问题 3：无维度覆盖度自检

**问题描述**：Prompt 没有要求 AI 检查四个维度是否都有覆盖。

**具体表现**：
- 用户输入 500 字，全是自我介绍和背景，完全没有提到 offer/need
- AI 仍然正常返回，offer 和 need 维度缺失
- 用户发布后发现"能提供的"和"具体需求"区域为空

**对产品目标的负面影响**：
- 个人主页结构性不完整
- 互赖网络的核心价值（offer↔need 匹配）无法发挥

#### 问题 4：无输入质量评估

**问题描述**：Prompt 对输入长度和信息密度没有差异化处理策略。

**具体表现**：
- 输入 10 个字："我是张三，做产品的" → AI 硬着头皮生成，blocks 可能只有 1 个，标签可能不准确
- 输入 2000 字：详细的人生经历 → AI 可能遗漏重要信息，或 blocks 过多导致前端展示混乱

**对产品目标的负面影响**：
- 极短输入生成质量差，用户可能认为"AI 不行"
- 极长输入信息丢失，用户需要手动补充

#### 问题 5：无标签可信度区分

**问题描述**：所有标签一视同仁，用户无法判断哪些是 AI"确定"的，哪些是"推测"的。

**具体表现**：
- AI 从"我在字节做产品"推断出「SaaS方向」——这个推断可能完全错误
- 用户看到标签云中有「SaaS方向」，但自己从未提过
- 用户困惑："为什么 AI 给我加了这个标签？"

**对产品目标的负面影响**：
- 用户对 AI 产生不信任感
- 需要逐一检查所有标签，增加纠偏工作量

#### 问题 6：简介生成 Prompt 过于简单

**问题描述**：`promptGenerateIntro` 只有一句话，没有结构化要求。

**具体表现**：
- AI 可能生成散文式简介，没有突出核心标签
- 不同调用之间风格差异大（有时正式、有时口语化）
- 简介和 blocks 之间可能出现信息重复

**对产品目标的负面影响**：
- 简介质量不稳定
- 用户需要反复编辑

### 2.2 AI 采访问题

#### 问题 1：无维度追踪

**问题描述**：AI 不知道已经覆盖了哪些维度，可能反复追问同一维度。

**具体表现**：
- 第 1 轮：AI 问"简单介绍一下自己" → 用户回答了自我介绍 + 背景
- 第 2 轮：AI 问"你之前在哪里上学/工作？" → 用户又回答了一遍背景信息
- 第 3 轮：AI 还在问背景相关的问题，完全没有触及 offer/need

**对产品目标的负面影响**：
- 信息获取效率低，6 轮内可能无法覆盖全部维度
- 用户感到 AI"没记住我说的话"，体验下降

#### 问题 2：追问策略无分层

**问题描述**："根据用户回答追问细节"太笼统，没有区分追问的层次和目的。

**具体表现**：
- 所有追问都是同一深度，没有"由浅入深"的层次感
- 用户回答很详细时，AI 还在问确认性问题（"你确定是在字节吗？"）
- 用户回答很简略时，AI 没有切换到深挖模式

**对产品目标的负面影响**：
- 对话节奏感差，用户可能感到无聊或被审讯
- 深层标签挖掘不足（H2 难以达标）

#### 问题 3：6 轮固定上限太死板

**问题描述**：无论信息饱和度如何，都强制聊满 6 轮或提前结束。

**具体表现**：
- 健谈的用户：6 轮后信息还远未充分，AI 被迫结束
- 沉默的用户：3 轮后已经无话可说，但 AI 还在继续问尴尬的问题
- 没有中间选项：用户不能"再聊 2 轮"，只能"结束"或"继续到 6 轮"

**对产品目标的负面影响**：
- 信息饱和度不可控，导致最终主页质量参差
- 用户感到被"强制"完成采访，产生反感

#### 问题 4：hiddenBlocks 无上下文累积

**问题描述**：每轮独立提取 hidden，没有去重和语义合并。

**具体表现**：
- 第 2 轮用户提到"其实我创业是因为在原公司待得不开心"
- 第 4 轮用户又说"离职前那段日子挺压抑的"
- AI 提取了两个 hiddenBlocks：「在原公司待得不开心」和「离职前很压抑」
- 这两个是同一信息的变体，但都被保留，造成冗余

**对产品目标的负面影响**：
- hidden 信息质量下降，重复信息干扰后续匹配
- 前端展示 hidden 列表时显得混乱

#### 问题 5：无收敛判断机制

**问题描述**：AI 不会主动判断"信息已足够"，也不会建议结束。

**具体表现**：
- 采访进行到第 5 轮，四个维度都已充分覆盖
- AI 仍然继续问第 6 个问题，而不是说"感谢您的配合，信息已经足够了"
- 用户可能在心里想"怎么还没完？"

**对产品目标的负面影响**：
- 采访体验冗长，用户可能在最后一轮前就已失去耐心
- 错过"恰到好处的结束"时机

#### 问题 6：无用户情绪感知

**问题描述**：AI 无法感知用户的参与度和情绪状态。

**具体表现**：
- 用户第 1 轮回答 200 字，第 2 轮 50 字，第 3 轮"就那样吧"
- AI 仍然按原计划继续提问，没有察觉到用户的敷衍
- 用户可能直接关闭页面

**对产品目标的负面影响**：
- 用户流失率上升
- 采访完成率下降

#### 问题 7：introSnippet 每轮重新生成

**问题描述**：没有"基于上一轮微调"的指令，可能导致简介片段前后不一致。

**具体表现**：
- 第 2 轮 introSnippet："前字节产品经理，现 AI 创业者..."
- 第 4 轮 introSnippet："一位有10年互联网经验的连续创业者..."
- 风格、人称、侧重点都在变
- 用户看到简介在"跳动"，感到不稳定

**对产品目标的负面影响**：
- 给用户"半成品"的感觉，降低对最终产品的期待

#### 问题 8：结束采访时的最终生成复用 promptExtractTags

**问题描述**：采访结束后用 `promptExtractTags` 生成最终内容，但未充分利用 interview 过程中积累的 hidden 信息的语义网络。

**具体表现**：
- hidden 信息被简单拼接进 systemPrompt
- AI 可能无法有效关联 hidden 信息和公开信息
- 例如：hidden 说"其实我想找合伙人是因为前合伙人背叛了"，公开信息说"正在寻找技术合伙人"——AI 没有利用这个因果关系来生成更有深度的简介

**对产品目标的负面影响**：
- hidden 信息的价值被浪费
- 最终主页质量未能达到"有 hidden 信息加持"应有的水平

#### 方案 B：Prompt + 预分析步骤

```
Round 1: 输入质量评估
```

### 3.2 AI 采访优化方案

---

#### 方案 A：纯 Prompt 增强（维度追踪 + 追问分层）

**改动内容：**

重写 `promptInterviewGuide`，增加以下结构：

1. **维度追踪状态**：每轮返回时输出 `dimensionCoverage`
2. **追问分层策略**：L1 确认 → L2 扩展 → L3 深挖
3. **信息饱和度评估**：输出 `saturation`（0-1）和 `suggestEnd`
4. **hidden 语义合并**：每轮提取后与历史 hidden 对比去重
5. **渐进式简介更新**：基于上一轮微调而非重写

**示例结构（供参考）：**

```
你是一位亲切的校友采访者，正在帮助实验中学2006届校友整理个人档案。

## 核心原则
1. 每次只问一个问题，语言自然亲切像老朋友聊天
2. 不是审讯，是帮助对方更好地认识自己
3. 如果用户表现出不耐烦（回答变短、敷衍），主动收敛进入 Summary

## 维度追踪（每轮必须输出）
你需要在内部维护以下状态，并在每轮输出中报告：

- self_intro：用户主动呈现的身份（覆盖度 0-3，0=未覆盖，3=充分）
- background：教育/职业经历（覆盖度 0-3）
- offer：能为校友提供的帮助（覆盖度 0-3，关键维度）
- need：需要的帮助（覆盖度 0-3，关键维度）

## 追问策略（按优先级选择）

【L1 确认层】当信息模糊或需要核实时使用
- 例："你刚才提到在字节做了3年，主要是负责哪块业务？"

【L2 扩展层】当某维度已覆盖但广度不足时使用
- 例："除了产品设计，你还有其他方面的专长可以分享给校友吗？"

【L3 深挖层】当发现潜在深层信息时使用
- 例："你从字节出来创业，当时是什么让你下定决心？那个触发点是什么？"

选择逻辑：
1. 优先覆盖未覆盖的维度（从 coverage=0 的开始）
2. 已覆盖维度中，选择 coverage 最低的进行 L2 扩展
3. 只有当用户透露了情绪/动机线索时，才使用 L3 深挖
4. 避免连续使用同一层级，保持对话节奏变化

## 信息饱和度评估

每轮结束时，评估整体信息饱和度（0-1）：
- 0.0-0.3：信息极少，需要大量追问
- 0.3-0.6：基本信息已获取，需要扩展和深挖
- 0.6-0.85：信息较充分，可以适度收敛
- 0.85-1.0：信息充分，建议结束采访

当 saturation ≥ 0.85 且所有维度 coverage ≥ 2 时，设置 suggestEnd = true。

## hidden 信息处理

从用户回答中提取 hidden 信息时：
1. 与历史 hidden 信息对比，语义重复的合并（保留更完整的版本）
2. 为每条 hidden 标注 `confidence`（用户明确表达 vs 推断）
3. 标注 `sourceRound`（第几轮提取的）

## 渐进式简介更新

introSnippet 应该基于上一轮的版本进行增量更新，而非重写：
- 保留已有信息
- 补充新获取的信息
- 调整语序使逻辑更流畅
- 保持人称和风格一致

## 输出格式

{
  "nextQuestion": "下一个问题（自然亲切，像一个老朋友在问）",
  "questionLevel": "L1|L2|L3",
  "targetDimension": "self_intro|background|offer|need",
  "dimensionCoverage": {
    "self_intro": { "covered": true, "depth": 1-3 },
    "background": { "covered": true, "depth": 1-3 },
    "offer": { "covered": false, "depth": 0 },
    "need": { "covered": true, "depth": 2 }
  },
  "saturation": 0.65,
  "suggestEnd": false,
  "hiddenBlocks": [{"content": "...", "confidence": "high|medium|low", "sourceRound": 3}],
  "extractedTags": [{"text": "...", "type": "..."}],
  "introSnippet": "基于目前信息的简介片段"
}
```

**预期效果：**
- 维度覆盖度显著提升，6 轮内覆盖全部维度的概率从当前的 ~60% 提升到 ~90%
- 追问质量提升，L3 深挖比例增加 → hidden 信息质量提升
- 信息饱和度评估使结束时机更合理

**优点：**
- 零代码改动（如果前端忽略新增的 dimensionCoverage/saturation 字段）
- 或最小代码改动（前端增加进度可视化）
- 采访体验更"智能"

**缺点：**
- Prompt 变长，token 消耗增加 50-80%
- 新增的 JSON 字段增加了输出复杂度，格式稳定性风险
- AI 可能"过度自信"地标记维度已覆盖（实际上信息很浅）

**实施难度：** ⭐⭐（低）

#### 方案 B：Prompt + 前端状态机（推荐）

**改动内容：**

在前端维护**采访状态机**，每轮将状态注入 systemPrompt：

```typescript
interface InterviewState {
  dimensionCoverage: {
    self_intro: { covered: boolean; depth: 0|1|2|3 };
    background: { covered: boolean; depth: 0|1|2|3 };
    offer: { covered: boolean; depth: 0|1|2|3 };
    need: { covered: boolean; depth: 0|1|2|3 };
  };
  informationDensity: number; // 0-1
  hiddenAccumulator: string[]; // 去重后的 hidden 信息
  askedQuestions: string[]; // 已问过的问题（避免重复）
  userEngagementScore: number; // 用户参与度 0-1（基于回答长度变化）
}
```

**前端每轮将 InterviewState 注入 systemPrompt：**

```
${promptInterviewGuide}

【采访状态报告】（由系统维护，请据此调整策略）
已覆盖维度：self_intro(depth=2), background(depth=1), offer(depth=0), need(depth=2)
信息饱和度：0.58
用户参与度：0.75（前3轮平均回答长度：120字, 80字, 150字）
已提取 hidden 信息（去重后）：["离职原因是与原公司理念不合", "希望找技术合伙人而非独自开发"]
已问过的问题（请勿重复）：["简单介绍一下自己", "你之前在哪里工作？", "为什么想出来创业？"]
当前是第 4 轮（动态上限：最多 8 轮，但饱和度 ≥ 0.85 时建议结束）
```

**动态轮数上限：**
```
基础上限：8 轮
如果 saturation ≥ 0.85 → 可以提前结束（AI 主动建议）
如果 userEngagementScore < 0.3 → 缩短至当前轮数 +1，加速收敛
如果某维度 coverage = 0 且已过半轮数 → 优先追问该维度
```

**UI 增强：**
- 在采访界面展示"维度覆盖进度条"（4 个小进度条，实时更新）
- 展示"信息饱和度"圆环图
- 当 `suggestEnd = true` 时，"结束采访"按钮高亮闪烁

**预期效果：**
- 采访效率提升 30-50%（相同信息量下轮数减少）
- 维度覆盖率接近 100%
- 用户参与度下降时自动收敛，减少流失

**优点：**
- 状态透明，AI 行为更可预测
- 动态轮数比固定轮数更灵活
- UI 进度可视化给用户"掌控感"

**缺点：**
- 代码改动较大（需要新增 InterviewState 管理逻辑）
- 状态注入增加了 systemPrompt 长度，token 消耗增加
- 需要设计合理的"深度评分"标准（前端如何判断 coverage depth？）

**实施难度：** ⭐⭐⭐⭐（较高）

#### 方案 C：Prompt + Interview Summary 步骤

**改动内容：**

在采访结束后、生成最终主页前，增加一个**Summary 确认步骤**：

```
采访结束（或用户点击「结束采访」）
  │
  ▼
【新步骤】Interview Summary
  ├── AI 基于全部 interview 信息生成总结：
  │   ├── 采访总结（3-5 句话概括获取到的核心信息）
  │   ├── 建议标签列表（含 AI 推断标签）
  │   ├── 建议简介（初稿）
  │   └── hidden 信息摘要（脱敏处理后的要点）
  │
  └── 用户确认界面：
      ├── [编辑总结]
      ├── [编辑标签]（增删改）
      ├── [编辑简介]
      ├── [管理 hidden 信息]（删除不想保留的）
      └── [确认并生成主页]
            │
            ▼
      基于用户确认后的内容，生成最终 blocks/tags/intro
```

这个步骤对应 PRD v1.5 中已有的"Creation Review"规格，但当前代码中**尚未实现**。

**预期效果：**
- 给用户"审稿"的机会，提升最终质量
- hidden 信息在生成前被用户确认/过滤，避免敏感信息意外泄露
- 减少发布后纠偏的需求

**优点：**
- 大幅提升最终内容质量
- 符合"用户是 Boss"的设计哲学
- PRD 中已有规格，产品方向一致

**缺点：**
- 增加一个步骤，可能让用户感到"又要多操作一步"
- UI 需要新增 Review 界面（大量前端工作）
- 与"快速录入"的路径不一致（快速录入没有 Review 步骤）

**实施难度：** ⭐⭐⭐⭐⭐（高）

#### 方案 D：全流程重构（采访 → 草稿 → Review → 发布）

**改动内容：**

将 AI 采访的流程对齐到 PRD 中定义的完整生命周期状态机：

```

  │
  ▼

  │
  ├── AI 逐维度提问 → 用户回答（最多 8 轮，动态收敛）
  │
  ├── Interview Summary（用户确认总结、标签、简介、hidden）
  │
  ├── 生成 Profile Draft（状态：draft）
  │
  ├── Creation Review（标签确认/拒绝、简介增删改、内容块编辑）
  │     ├── 增量标签用琥珀金高亮

  └── Published 庆祝状态
        ├── Toast「你的主页已上线」
        ├── [查看我的主页] [继续完善] [返回基地]
        └── 图片异步生成中...
```

**与当前代码的差异：**
- 当前：采访结束 → 直接生成 → 发布（缺少 Summary + Review）
- 重构：采访结束 → Summary → Review → 发布 → 庆祝

**预期效果：**
- 完整对齐 PRD 规格，用户体验闭环
- 内容质量最高（多轮确认）
- 产品专业度提升

**优点：**
- 最完整的用户体验
- 产品质量最高
- 与 PRD 完全一致

**缺点：**
- 工作量最大（需要新增 Summary 页面、Review 页面、庆祝页面）
- 流程变长，可能降低"快速完成"的爽感
- POC 阶段可能过度设计

**实施难度：** ⭐⭐⭐⭐⭐（最高）

### 3.3 通用优化方案

#### 方案：提示词优化助手

**改动内容：**

在 TuningPanel 增加一个按钮：**"🪄 用 AI 优化此 Prompt"**

点击后，调用 `/api/chat`，systemPrompt 为：

```
你是一位顶级的 AI Prompt 工程师，擅长为中文 AI 应用设计高质量的系统提示词。

请对下面的 Prompt 进行专业优化，输出优化后的版本。

优化原则：
1. 结构清晰：使用明确的层级标题分隔不同指令
2. 边界完整：覆盖正常输入、边界输入、异常输入的处理
3. Few-shot：如果适用，提供输入→输出的示例
4. 自检机制：要求 AI 在输出前进行自我检查
5. 输出稳定：使用 JSON Schema 或严格的格式规范
6. 角色鲜明：让 AI 清楚自己的角色和任务

请输出：
- 优化后的完整 Prompt
- 优化说明（逐条解释改动原因）
```

**预期效果：**
- 用户（产品经理）可以自助优化 Prompt，无需写代码
- 验证假设 H4：Prompt 修改到生效 < 5 分钟

**优点：**
- 极大降低 Prompt 调优门槛
- 利用更强的模型（DeepSeek-V4-pro）做 Prompt 工程
- 有趣的产品特性，可作为卖点

**缺点：**
- 需要额外 API 调用（但只需在 TuningPanel 使用，频率低）
- AI 优化的 Prompt 可能过长，需要用户手动精简
- 可能产生"过度工程化"的 Prompt

**实施难度：** ⭐⭐（低）

#### 方案：Temperature 场景化

**改动内容：**

当前所有场景共用 `state.config.temperature`（默认 0.7）。建议为不同场景配置不同默认值：

| 场景 | 推荐 Temperature | 理由 |
|------|-----------------|------|
| 快速录入 - 标签提取 | 0.3-0.5 | 需要稳定、可预测的结构化输出，减少格式抖动 |
| 快速录入 - 简介生成 | 0.5-0.7 | 平衡创意和可读性 |
| AI 采访 - 对话生成 | 0.7-0.9 | 需要自然、多变的对话风格，避免机械感 |
| AI 采访 - 标签提取 | 0.5-0.7 | 采访过程中提取的标签也需要一定稳定性 |
| 图片生成 Prompt | 0.5-0.7 | 图片模型可能不受 temperature 直接影响，但保持中等 |

**实现方式：**
- 在 TuningPanel 中，temperature slider 下方增加场景细分，或
- 后端自动根据调用场景选择 temperature（覆盖用户设置）

**预期效果：**
- 标签提取稳定性提升（减少 JSON 解析失败）
- 采访对话更自然、不机械
- 整体输出质量提升

**优点：**
- 技术简单，效果直观
- 不增加用户操作复杂度

**缺点：**
- 用户失去全局 temperature 控制权（除非提供"高级模式"覆盖）
- 不同模型的 temperature 敏感度不同，需要针对具体模型微调

**实施难度：** ⭐（极低）
